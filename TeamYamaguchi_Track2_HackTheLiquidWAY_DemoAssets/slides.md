# 方言話者は音声AIを使えない

**Hack the Liquid WAY 2026 — Track 2 / チーム山口**

---

## Slide 1: 問題 / ユースケース

- 日本には多様な方言（関西弁、九州弁、東北弁 etc.）が存在
- 既存の音声認識・音声AIは標準語前提 → **方言話者が弾かれる**
- 特に高齢者・地方在住者にとって「声で操作できない」は深刻
- 例: スマートスピーカー、音声入力、カーナビ、コールセンターAI

> 関西弁で話すと、スマートスピーカーが理解してくれない。
> 標準語に「翻訳」しながら話すのは疲れる。

---

## Slide 2: なぜ LFM？

**LFM2.5-Audio-1.5B-JP だから 1モデルで音声→音声変換できる**

| | クラウドLLM（一般的なAPI構成） | LFM2.5-Audio-1.5B-JP |
|---|---|---|
| 音声→音声 | ASR + LLM + TTS の3段構成 | **1モデル・1推論** |
| エッジ動作 | 不可（クラウド必須） | **エッジデバイス上で動作** |
| レイテンシ | 高（ネットワーク往復） | **低（オンデバイス）** |
| プライバシー | 音声データがクラウドへ | **完全ローカル処理** |
| カスタマイズ | 音声Fine-tune不可 | **LoRAで43MBのアダプタ** |

テキストトークンと音声フレームを同一モデルで生成 → LFMのアーキテクチャがEnd-to-Endの音声処理を可能にする

---

## Slide 3: アプローチ・アーキテクチャ

**目標パイプライン（音声→音声）**:
```
[方言音声入力 16kHz] → [LFM2.5-Audio-1.5B-JP + LoRA (43MB)] → [標準語音声]
```
※ 今回のデモは音声出力の安定化が間に合わず、テキスト出力 + edge-tts 読み上げで実施

**学習データ**:
- 出典: `federerjiang/dialect.osaka` + `federerjiang/dialect.kumamoto`（HuggingFace）
- 方言音声×方言テキストのペアをベースに、Qwen3-32B で標準語テキストを自動生成
- 学習ペア: **方言音声 → 標準語テキスト**（大阪弁 + 熊本弁 計2,600件）

**LoRA 設定**: r=16, alpha=32 / 対象層: q/k/v/out_proj, w1/w2/w3 / 訓練パラメータ 0.76% / RTX 5090で学習

**工夫**:
- 学習データ末尾に `<|text_end|>` を付与 → 音声フレームへの切り替えタイミングを学習
- Greedy decoding (top_k=1): 最も確率の高いトークンを選択 → 再現性のある安定した出力
